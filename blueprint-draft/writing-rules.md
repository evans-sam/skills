# Writing Rules

These rules apply to all blueprint output — both the working doc and the final Notion page.

## Tone

- Factual, concise, decided.
- No emoji.
- No cheerful AI filler ("Great news!", "This exciting feature...", "Let's dive in!").
- No hedging above the fold. "We will," not "we might consider."

## Language

- Above the fold: "We will..." / "This service does..." / "The endpoint accepts..."
- Below the fold (scratchpad): "We might consider..." / "Open question:..." is acceptable.

## Self-Check

If you write any of these above the fold, rewrite or move to Risks/Open Questions:
- "maybe"
- "potentially"
- "we could consider"
- "one option would be"
- "we might"
- "we propose" (did you decide or not?)

## Structure

- Diagrams encouraged. Prose walls discouraged.
- If you're writing paragraph after paragraph describing data flow, use a diagram instead.
- Short paragraphs. Anyone at the company should be able to read the blueprint in under 10 minutes.

## Estimates Table

- Every row needs a meaningful description — not just a title.
- Every row maps to work described in Technical Implementation.
- Every item in Technical Implementation maps to an estimate row.
- Include testing time as its own row or baked into relevant rows.
